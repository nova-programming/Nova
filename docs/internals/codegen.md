# Nova Native Codegen Internals (`stdlib/codegen.nv`)

The `codegen.nv` module is responsible for traversing the Abstract Syntax Tree (AST) generated by `parser.nv` and outputting raw x86 assembly (specifically structured for NASM syntax and the MinGW/GCC linker) to produce a standalone Windows executable.

## State Management

The `CodegenState` struct tracks:
* `assembly`: A list of strings representing the generated assembly instructions.
* `data_section`: A list of strings representing static string literals and format strings.
* `local_vars`: A dictionary (hash map) mapping local variable names to their memory offset relative to the `ebp` stack pointer.
* `local_offset`: An integer tracking the total size of local variables on the stack frame.
* `label_counter`: An integer used to generate unique jump labels for control flow (loops, if/else).

## Pass 1: Variable Scanning (`scan_vars`)

Before compiling a block of statements (like a function body), `scan_vars` recursively walks the AST of that block. Its sole purpose is to locate `Assignment` nodes and register them in the `local_vars` map, updating `local_offset`. 

**Critical Edge Case:** `scan_vars` does *not* recursively scan inside `@raw` (`RawBlock`) nodes. Any variables introduced *exclusively* within a `@raw` block will not be assigned stack space and will crash the compiler when referenced. Variables must be initialized to a dummy value (e.g., `cmdline = 0`) *before* the `@raw` block to guarantee stack frame allocation.

## Pass 2: Instruction Generation

The code generation is broken into two main functions: `compile_stmt` (statements) and `compile_expr` (expressions). The output strategy relies entirely on the **x86 cdecl stack calling convention**.

### Stack-Based Expression Evaluation
All expressions push their resulting value onto the x86 stack.
* `compile_expr(Integer)`: `push [value]`
* `compile_expr(Variable)`: Looks up the variable in `local_vars`. Generates `mov eax, [ebp - offset]` followed by `push eax`.
* `compile_expr(BinOp)`: 
   1. Recursively compiles left operand (pushes to stack).
   2. Recursively compiles right operand (pushes to stack).
   3. Pops right operand into `ebx`.
   4. Pops left operand into `eax`.
   5. Performs the math (e.g., `add eax, ebx`) and pushes the result back to the stack.

### Statements
* **Function Definitions (`def`):** Emits standard prologue (`push ebp`, `mov ebp, esp`, `sub esp, [local_offset]`). Compiles body. Emits epilogue (`mov esp, ebp`, `pop ebp`, `ret`).
* **Assignments (`=`):** Compiles the RHS expression, pops it into `eax`, and stores it via `mov [ebp - offset], eax`.
* **If/Else & While Loops:** Compiles the condition expression, pops it, compares it against `0`, and conditionally jumps using globally unique labels generated by `next_label()`.
* **Raw Blocks (`@raw`):** Used to bypass native type checking or to allow pointer/memory arithmetic via `.value_byte` for reading characters. It compiles standard statements exactly like regular Nova.

## Function Calls and FFI (Foreign Function Interface)
Function calls in `compile_expr` are processed by compiling and pushing all arguments in **reverse order** (standard `cdecl`). 
If a function call matches a globally whitelisted C-standard library function (e.g., `_printf`, `_fflush`, `_malloc`, `_GetCommandLineA`), it is compiled natively via `call _Func`. 

### Strings and Allocations
Nova natively leverages `malloc` for allocations:
* String literals are stored in the `.data` section and pushed as `offset string_label`.
* Explicit `alloc(length)` calls compile to a direct `_malloc` call, returning a raw memory pointer.
* List access (e.g. `base[index]`) automatically detects via inference whether it is indexing a C-string or a List struct. String indexing extracts a byte (`movzx`) and allocates a new 2-byte null-terminated C-string to wrap the character, making standard string manipulations intuitive in Nova while retaining native memory layouts.
